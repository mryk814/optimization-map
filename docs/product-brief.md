# Product Brief

## コンセプト

Optimization Map は、最適化を「課題から入れる」ための教育的ナビゲーションシステムです。

既存の教材・ツールは、LP、MILP、Convex Optimization、Bayesian Optimization、RL、Optimal Control など、それぞれの領域では強い一方で、分野横断の入口が弱いです。Optimization Map は、次の流れを一つの知識ベースとしてつなぎます。

```txt
現実課題
  → 決定変数
  → 目的関数
  → 制約
  → 不確実性 / 時間性 / ブラックボックス性
  → 問題クラス候補
  → モデリング方針
  → アルゴリズム候補
  → ソルバー / ライブラリ候補
```

## 主要ユーザー

### 1. 初学者

- 最適化の用語が多すぎて地図がほしい
- LP / MILP / Convex / NLP / BO / RL などの関係がわからない
- 自分の課題をどう定式化すればよいかわからない

### 2. 実務家

- 配送、スケジューリング、価格、実験、リソース配分などの課題がある
- どの問題クラス・アルゴリズム・ソルバーが候補になるか知りたい
- 厳密解、近似解、計算時間、説明責任、制約違反リスクを判断したい

### 3. 研究者 / 上級者

- 隣接領域との関係を調べたい
- 問題クラス、緩和、分解、双対、微分可能化などの関係を見たい
- 教材・論文・ソフトウェアを横断的に探したい

## プロダクトが解くギャップ

| ギャップ | 既存リソースの傾向 | Optimization Map の方針 |
|---|---|---|
| 全体地図がない | 教科書や講義は分野別 | 問題クラスを知識グラフとして可視化 |
| 課題から入れない | すでに問題名を知っている前提 | 診断フローで候補を提示 |
| 似た概念の比較が弱い | 個別説明はあるが比較 UX が少ない | `confused_with` と比較ビューを持つ |
| ソルバー選定が分断 | ソルバー docs は実装寄り | 問題クラス → アルゴリズム → ソルバーを接続 |
| 難しさの理由が見えない | NP-hard / nonconvex などで止まりがち | 難しさを分解して表示 |

## 基本設計原則

### 多軸分類

問題クラスは相互排他的なカテゴリではありません。たとえば SDP は Convex Optimization でもあり Conic Optimization でもあり、Combinatorial Optimization の relaxation としても使われます。したがって、単一階層ではなく、ファセットと関係グラフで表現します。

### 診断は説明可能にする

診断結果は「MILP です」とだけ返さず、なぜそう判断したかを出します。

```txt
候補: MILP
理由:
- 決定変数に binary / integer がある
- 目的関数と制約が線形で書けそう
- 厳密解または最適性ギャップが重要
```

### 推薦は断定しない

ソルバーやアルゴリズム推薦は「唯一の正解」ではなく、候補・前提・注意点として提示します。

### 教育 UX を優先する

最初の価値は、解くことではなく、理解・比較・探索・モデル化支援です。ソルバー実行は後回しでよいです。

## MVP での成功条件

- 主要問題クラス 15〜18 個がカードとして定義されている
- 分類軸 24〜28 個が定義されている
- 関係グラフが 80 本以上ある
- 10〜20 個の現実課題を診断できる
- 診断結果が「候補」「理由」「注意点」「次に読むべきもの」を返す
- 似ている概念を 5 組以上比較できる
- 主要ソルバー / ライブラリ 15〜20 個が対応付けられている

## Non-goals

- 独自最適化ソルバーの開発
- すべての最適化論文の収集
- 全ソルバーの厳密な性能比較
- 理論証明の完全な教材化
- LLM 任せの自動 taxonomy 生成
